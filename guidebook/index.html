Introduction

KeywordFlow is a powerful test automation framework which supports Keyword Driven Development and BDD .It contains over 170 predefined step definitions using Gherkin language that allows you to start the testing process and write feature files in your project right away (Appendix A). It explains the behavior of the application in a simple English text using Gherkin language. QA can mix, customize step definitions to create an infinite variety of tests.

This feature-rich framework enables rapid automation scripts development and helps to increase automation test coverage in less time, thus improving product quality. It was developed by Exadel Inc. This tutorial starts with a basic introduction and introduces you to the advanced features with examples.

Areas where KeywordFlow can be applied

There are a number types of testing that can be done with KeywordFlow. You can do API functional, API integration, UI functional, UI end-to-end.

Audience

This tutorial is targeted for first-time learners, as well as moderate users. Beginners will get a fair understanding about KeywordFlow and its usage, and others will be able to refresh their knowledge on this subject.

Prerequisites

This tutorial requires that the readers have some preliminary knowledge of the automation testing process, along with identification of the correct GUI element on a web page. In the later sections of this tutorial, we will discuss integration with build tools, version control systems. It is assumed that the required tools are installed and configured on the system and the reader is familiar with those tools.

First scenario implementation

Let's imagine that we are faced with the task of automating such a business scenario:

Scenario: Verify search functionality on wiki page works correctly

Given: I’m on wiki home page

When I search by some key phrase

Then I’m redirecting to a page with searched result

Now we can try to cover this scenario with steps that already exist in our framework. Such an approach is usually called a keyword driven in automation testing.

Firstly we need to open a VSCode and create a new wiki-test.feature file in src/features/ui-features directory:

When wiki-test.feature file is created we can start working on a test content:

@example

Feature: Test wiki search

    Description of the test feature - should demonstrate Search work.

    Scenario: Verify search functionality works correctly

We’ve created a first feature description and gave a name to our first test scenario.

Now we need to add a first user actions step that navigates us to the wiki start page and make a search actions:

Given User navigates to "https://www.wikipedia.org/"

        When User enters "Isaac Newton" in [Search field] "#searchInput"

            And User clicks [Search button] "button[type=submit]"

After we’ve performed action steps we need to complete our test with some verifications:

Check the page title and correct URL path after search is completed

Then [Page title] "#firstHeading" text is equal to "Isaac Newton"

            And Page URL contains "Isaac_Newton"

Finally our first test is ready:

@example

Feature: Test wiki search

    Description of the test feature - should demonstrate Search work.

 

    Scenario: Verify search functionality works correctly

        Given User navigates to "https://www.wikipedia.org/"

        When User enters "Isaac Newton" in [Search field] "#searchInput"

            And User clicks [Search button] "button[type=submit]"

        Then [Page title] "#firstHeading" text is equal to "Isaac Newton"

            And Page URL contains "Isaac_Newton"

And all we need to do is to run it with execute the command:

npm run test-setup

After the test is completed we can analyze results.

Execute command to open allure reporter:

npm run allure:open

Key points to remember

  • Cucumber executes your .feature files, and those files contain executable specifications written in a language called Gherkin.
  • In Gherkin, each line that isn't blank has to start with a Gherkin keyword, followed by any text you like. The main keywords are:
  • Feature         
  • Scenario
  • Given, When, Then, And, But (Steps)
  • Background
  • Scenario Outline
  • Examples
  • There are a few extra keywords as well:
  • """ (Doc Strings)
  • | (Data Tables)
  • @ (Tags)
  • # (Comments)
  • On the lines following a Feature, Scenario, Scenario Outline or Examples you can write anything you like, as long as no line starts with a key or a keyword.

Feature Syntax

  • Feature files should be located under the ‘features’ folder with ‘.feature’ extension.
  • A feature contains a list of scenarios.
  • Every scenario consists of a list of steps, which must start with one of the keywords Given, When, Then, But or And

@example

Feature: Test wiki search

    Description of the test feature - should demonstrate Search work.

   

    @scenario1

    Scenario: Verify search functionality works correctly

        Given User navigates to "https://www.wikipedia.org/"

        When User enters "Isaac Newton" in [Search field] "#searchInput"

            And User clicks [Search button] "button[type=submit]"

        Then [Page title] "#firstHeading" text is equal to "Isaac Newton"

            And Page URL contains "Isaac_Newton"

@example -> Feature tag

Feature: -> Keyword containing Feature description

Scenario -> Keyword containing Scenario description

@scenario1 -> Scenario tag

Short Access: (Ctrl) + (Mouseover on particular line) -> Redirected to the particular step-definition

Background

Occasionally you'll find yourself repeating the same Given steps in all of the scenarios in a feature file. Since it is repeated in every scenario it is an indication that those steps are not essential to describe the scenarios, they are incidental details. You can literally move such Given steps to the background by grouping them under a Background section before the first scenario:

@example

Feature: Test wiki search

    Description of the test feature - should demonstrate Search work.

 

    Background:

          Given User navigates to "https://www.wikipedia.org/"

   

    @scenario1

    Scenario: Verify search functionality works correctly       

        When User enters "Isaac Newton" in [Search field] "#searchInput"

            And User clicks [Search button] "button[type=submit]"

        Then [Page title] "#firstHeading" text is equal to "Isaac Newton"

            And Page URL contains "Isaac_Newton"

   

    @scenario2

    Scenario: Verify Privacy Policy link navigation in Footer

         When User clicks [Privacy Policy link] "#footer-places-privacy"

       Then [Page title] "#firstHeading" text is equal to "Privacy Policy"

Scenario Outlines

When you have a complex business rule with several variable inputs or outputs you might end up creating several scenarios that only differ by their values.

A Scenario Outline section is always followed by one or more Examples sections, which are a container for a table. The table must have a header row corresponding to the variables in the Scenario Outline steps. Each of the rows below will create a new Scenario, filling in the variable values.

@example

Feature: Test wiki search

    Description of the test feature - should demonstrate Search work.

 

  Background:

    Given User navigates to "https://www.wikipedia.org/"

  Scenario Outline: Verify search functionality works correctly

    When User enters <input> in [Search field] "#searchInput"

      And User clicks [Search button] "button[type=submit]"

    Then [Page title] "#firstHeading" text is equal to <title>

      And Page URL contains <url>

    Examples:

      | input        | title           | url             |

      | Isaac Newton | Isaac Newton    | Isaac_Newton    |

      | Einstein     | Albert Einstein | Albert_Einstein |

Tags and its usage

Features can be selectively run based on tags. Tags are written above Feature or Scenario using ‘@’ symbol. Tags can also be used on scenarios, to run selective scenarios.

Run Commands

npx wdio wdio.local.conf.js -> run all specs set in wdio.local.conf.js

npx wdio wdio.local.conf.js --tag=@example -> run specs set in wdio.local.conf.js only with tag @example

npm run test-local -> predefined script command described in package.json - run specs set in wdio.local.conf.js only with tag @debug

Adding New Steps

The predefined step definitions allow you to do multiple common things. All available UI step definitions are stored in a library \node_modules\keywordflow-wdio-js-lib\step-definitions and divided into action-step-definitions, verification-step-definitions.

But you might need extra steps which are project specific or just better aligned with your goals and you will find custom action-step-definitions and verification-step-definitions inside your project repo src\features\step-definitions as well as API step definitions: api-step-definitions and api-verification-step-definitions. And all these steps might be modified and a new one added as well.

Appendix A: List of predefined steps

Vocabulary

.* - Any number of words should be set. Minimum 1 symbol.  “ ” are forbidden

“locator” - css or xPath selector of necessary web element need to be set inside “ ”

“text” - Expected text need to be set inside “ ”

“parentLocator” - Css selector of Parent web element need to be set inside “ ”

“parentText” - Expected text of parent web element need to be set inside “”

"childLocator" - Css selector of Child inside Parent web element need to be set inside “”

“childText” - Expected text of child web element need to be set inside “”

"dropdownLocator" - Css selector of Dropdown web element need to be set inside “”

"itemLocator" - Css selector of Item inside Dropdown web element need to be set inside “”

“url” - URL with protocol (http or https) should be set inside “”.  Or may  use PAGE_NAME property defined in user-data.js

“fileName” - name of file to be uploaded. Should be placed only in /features/test-data/ folder of your current project. Should contain file name and filename extension

number - an expected number should be set without “”.

“downloadedFileName” – name of file that we expect to be downloaded. Should contain file name and filename extension.

“hash” – file hash. Should have the next format “HASH:value”. Where “HASH:” is a constant part. “value” – should be changed with appropriate hash.
